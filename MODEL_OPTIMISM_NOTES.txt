OSRS Flip Advisor - model optimism notes
=======================================

Purpose
-------

This note documents where the current recommendation model can still be too
optimistic, why that matters for real Grand Exchange flipping, and what changes
should be made next. The app is already much better than a raw margin scanner,
because it uses historical distribution targets, tax, risk scoring, fill
probability, drift, and persistent open-order tracking. The remaining issues are
mostly about making the modeled profit closer to what happens when an offer sits
for hours, partially fills, or has to be exited at an unattractive price.


1. Fill probability is probably too high
---------------------------------------

Current behavior:

The model uses a two-tail Student-t survival probability for fill odds. That is
conservative compared with a normal distribution in the sense that it has fatter
tails, but it is optimistic for GE fills because a buy and a sell are one-sided
events:

- A buy offer below fair value fills only if the market moves down to that bid.
- A sell offer above fair value fills only if the market moves up to that ask.

A two-tail probability counts both directions. For a one-sided fill, that can
roughly double the chance of filling.

Why it matters:

Expected weekly profit is directly multiplied by fill probability. If the fill
odds are too high, the app can overstate weekly profit, understate time stuck in
offers, and suggest prices that look mathematically good but do not realistically
cycle enough times per week.

Recommended change:

Replace the current two-tail probability with side-aware one-tail probabilities:

- buyFillProbability = P(price <= entryTarget)
- sellFillProbability = P(price >= exitTarget)

The Student-t distribution can still be used, but the survival value should be
one-sided for a single entry or exit event. The UI should also label these as
"entry fill chance" and "exit fill chance" rather than one generic probability.


2. Open sell-order advice can ignore break-even
-----------------------------------------------

Current behavior:

The open-order panel can tell the user to lower a stuck sell order toward the
drift-adjusted model exit. That is useful for freeing capital, but it does not
always check the tracked average cost of the held position first.

Why it matters:

If the player bought an item before shutting down the server, or if tracking is
incomplete, the model exit might be below the real break-even price. In that
case, "lower ask to X" can accidentally become "lock in a loss" without making
that decision explicit.

Recommended change:

When the app has an average cost from tracked positions or from manual recovery
input, sell guidance should calculate:

- breakEvenSell = the minimum sell price where sellPrice - tax >= cost
- modelExit = distribution sell target or drift-adjusted sell target
- lossAvoidanceTarget = max(breakEvenSell, modelExit)

If the model recommends selling below break-even, the UI should say that it is a
loss-exit decision and show the estimated loss after tax. Do not present it as a
normal reprice.


3. Open-order tax math should use the real GE tax cap
----------------------------------------------------

Current behavior:

Some browser-side open-order calculations approximate net sell value as
sellPrice * 0.98. That is close for most items, but it does not match the actual
tax model, which is floor(2%) capped at 5,000,000 gp per item.

Why it matters:

For very expensive items, the 5m cap means net proceeds are higher than a flat
98% calculation. The app may be too pessimistic on high-value items and may tell
the user to cancel or avoid a reprice even when the capped-tax math leaves a
reasonable margin.

Recommended change:

Use the same tax helper everywhere:

tax = min(floor(sellPrice * 0.02), 5,000,000)
netSell = sellPrice - tax

The browser already has a matching clientTax helper. All open-order and recovery
calculations should use that helper instead of multiplying by 0.98.


4. Adaptive entry fill time is based on the wrong quantity
---------------------------------------------------------

Current behavior:

The model estimates entry fill time using a pre-sizing quantity based mainly on
slot budget and item fair value. Later, the actual displayed quantity can be
reduced by liquidity, buy limit, risk score, and loss budget.

Why it matters:

The app can reject an item as "cannot fill fast enough" even though the smaller
final position would have filled within the target time. It can also display a
fill-time estimate that does not correspond to the final quantity shown to the
player.

Recommended change:

Calculate fill time after final position sizing, or run sizing in two passes:

1. Build a tentative opportunity and final quantity.
2. Solve adaptive entry depth using that final quantity.
3. Recompute profit, fill time, expected value, and risk from the adjusted entry.


5. Weekly profit still assumes smooth recycling
-----------------------------------------------

Current behavior:

Weekly profit is based on modeled units per cycle and cycles per week. It does
consider buy limits and EV, but it still treats cycling as relatively smooth.

Why it matters:

In real flipping, a slot can be blocked by:

- partial fills
- a buy that fills but then cannot sell
- a sell that must be repriced
- user sleep/offline time
- item limit exhaustion
- capital trapped in a bad position

These frictions lower real weekly profit versus the model.

Recommended change:

Add a slot-occupancy discount. A simple version:

effectiveCyclesPerWeek = 168 / (entryFillHours + exitFillHours + reviewDelayHours)

Then cap weekly units by both buy limit and effective cycles. For pinned items,
track real time spent in each state and use that as a learned penalty.


6. Trend handling can still chase temporary moves
------------------------------------------------

Current behavior:

The adaptive horizon shortens the half-life during a trend and drift-aware EV
projects fair value forward. This helps avoid stale prices, but a short burst can
still look like a sustainable trend if it dominates the recent sample window.

Why it matters:

The app can credit an uptrend too quickly and suggest an exit that depends on
the trend continuing. In OSRS, updates, content releases, streamer attention,
and thin liquidity can reverse quickly.

Recommended change:

Require trend confirmation across at least two windows:

- short window drift, such as 6 to 12 hours
- long window drift, such as 48 to 72 hours

If they disagree, reduce confidence and position size. If the item is newly
added, recently patched, or has limited history, cap trend credit more harshly.


7. Manual recovery should become a first-class risk workflow
-----------------------------------------------------------

Current behavior after this update:

The dashboard now includes a manual Recovery search. It lets the player search
by item name or id, enter an optional buy price and quantity, and see:

- current low and high
- model sell target
- break-even sell price
- quick-exit profit or loss after tax
- relist target that will not go below break-even when a buy price is supplied
- option to pin the item back into a GE slot as a recovery plan

Why it matters:

This addresses the specific failure mode where the player buys an item, shuts
down the server, loses the pinned state or fill context, and later has no clear
sell target. The recovery flow should prevent panic listing below break-even
unless the user intentionally chooses a loss exit.

Recommended future changes:

- Save recovery searches locally so recent manual lookups survive browser close.
- Let the user attach manual holdings to tracked positions.
- Add an explicit "I own this" button from recovery search that creates a holding
  even when RuneLite fill tracking missed the buy.
- Show "loss avoidance target", "fast exit target", and "patient model target" as
  separate choices instead of one blended target.


Priority order
--------------

1. Switch fill probability from two-tail to one-tail side-aware probabilities.
2. Make sell reprice advice break-even aware.
3. Replace flat 0.98 tax approximations with the capped tax helper everywhere.
4. Recompute adaptive entry fill time after final quantity sizing.
5. Add slot-occupancy discounts to weekly profit.
6. Add trend confirmation and stronger new-item penalties.
7. Expand manual recovery into persistent manual holdings.
